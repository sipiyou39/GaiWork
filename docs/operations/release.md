# Doudou Code releases and automatic updates

Doudou Code already includes a complete desktop updater based on `electron-updater`. Sparkle does not need to be added separately: on macOS, `electron-updater` uses the Squirrel.Mac update format generated by `electron-builder`.

The fork publishes stable macOS builds from `.github/workflows/release.yml` to the public GitHub repository `sipiyou39/GaiWork`.

## User experience

Friends install Doudou Code once from the signed and notarized DMG. Later releases are discovered from inside the application:

- a first background check runs 15 seconds after startup;
- another check runs every four minutes while the app remains open;
- **Check for Updates…** is also available from the macOS application menu;
- Doudou Code never downloads or restarts without the user choosing to do so;
- the update control first downloads the release, then offers to restart and install it.

The update feed is embedded in packaged builds as `app-update.yml`. It points to GitHub Releases and uses the canonical `latest-mac.yml` manifest published alongside the DMG and ZIP files.

## Why signing is mandatory

macOS auto-update requires every version in the chain to be signed with the same Developer ID identity. Notarization also prevents friends from receiving an unidentified-developer warning on first launch.

Local unsigned DMGs remain useful for development, but the public release workflow deliberately fails when signing credentials are missing instead of publishing a build that cannot update reliably.

## One-time Apple setup

An Apple Developer Program membership is required.

1. In Certificates, Identifiers & Profiles, register an explicit macOS App ID for Doudou Code's final bundle ID:

   ```text
   io.github.sipiyou39.doudoucode
   ```

   The visible App ID description can also be `Doudou Code`. This bundle ID must remain unchanged after the first public Doudou Code release so every future version stays in the same update chain. Existing GaiWork development data is recovered through local compatibility fallbacks, but an old GaiWork application bundle cannot auto-update across this final bundle-ID change.

2. Create a **Developer ID Application** certificate for the team.
3. Import it into Keychain Access, export the certificate and private key as a password-protected `.p12`, then encode it:

   ```bash
   base64 -i DoudouCode-Developer-ID.p12 | pbcopy
   ```

4. In App Store Connect, create an API key that can submit notarization requests. Download its `.p8` file and record its key ID and issuer ID.
5. Add these repository secrets under **Settings → Secrets and variables → Actions**:

   | Secret             | Value                                   |
   | ------------------ | --------------------------------------- |
   | `CSC_LINK`         | Base64 contents of the exported `.p12`  |
   | `CSC_KEY_PASSWORD` | Password used when exporting the `.p12` |
   | `APPLE_API_KEY`    | Raw contents of the `AuthKey_….p8` file |
   | `APPLE_API_KEY_ID` | App Store Connect API key ID            |
   | `APPLE_API_ISSUER` | App Store Connect issuer ID             |

The same secrets can be configured from an authenticated GitHub CLI:

```bash
gh secret set CSC_LINK
gh secret set CSC_KEY_PASSWORD
gh secret set APPLE_API_KEY < AuthKey_KEYID.p8
gh secret set APPLE_API_KEY_ID
gh secret set APPLE_API_ISSUER
```

The basic release does not require Clerk, Cloudflare, Vercel, a provisioning profile, or Doudou Code's release GitHub App. Optional Clerk passkey entitlements remain supported by the build script, but they are activated only when their dedicated configuration is present.

## Publishing a release

All four releasable package manifests must carry the same semantic version. Prepare a clean release commit on `main`:

```bash
version=0.1.0
node scripts/update-release-package-versions.ts "$version"
vp check
vp run typecheck
vp test
git add apps/server/package.json apps/desktop/package.json apps/web/package.json packages/contracts/package.json
git commit -m "chore(release): prepare Doudou Code v$version"
git push origin main
```

Then publish by tag:

```bash
git tag "v$version"
git push origin "v$version"
```

Alternatively, open **Actions → Doudou Code Desktop Release → Run workflow** and enter the same version. The selected commit must belong to `origin/main`, and its package versions must already match the input.

The workflow:

1. runs `vp check`, `vp run typecheck`, and `vp test`;
2. builds signed and notarized Apple Silicon and Intel applications;
3. creates a DMG and ZIP for each architecture;
4. merges both architecture entries into `latest-mac.yml`;
5. publishes a public GitHub Release named `Doudou Code vX.Y.Z`;
6. marks it as the latest stable release.

Never reuse a version number. `electron-updater` only offers a release whose semantic version is newer than the installed version.

## Required release assets

Do not delete individual assets from a published release. A valid macOS release contains:

- one `.dmg` for Apple Silicon;
- one `.dmg` for Intel;
- one `.zip` for Apple Silicon;
- one `.zip` for Intel;
- the matching `.blockmap` files;
- `latest-mac.yml`, containing entries for both architectures.

The DMGs are for the first installation. Squirrel.Mac downloads the ZIP selected by `latest-mac.yml` for subsequent in-place updates.

## Release smoke check

After the first publication:

1. download the correct DMG from the GitHub Release on a Mac that has never run the development build;
2. drag Doudou Code to `/Applications` and launch it;
3. confirm **About Doudou Code** shows the published version;
4. publish a patch release with a higher version;
5. use **Doudou Code → Check for Updates…**;
6. download, restart, and confirm the new version without reinstalling the DMG;
7. confirm conversations, settings, and companion positions are preserved.

## Troubleshooting

### Updates are disabled

Automatic updates only run in a packaged production build that contains `app-update.yml`. They are intentionally disabled in the development app and in unpackaged builds.

### The app reports no update

Check that:

- the GitHub Release is public and not a draft;
- its version is greater than the installed version;
- it is marked as the latest stable release;
- `latest-mac.yml` and both ZIP files are present;
- the manifest URLs and SHA-512 hashes correspond to the uploaded files.

### The workflow fails before packaging

The workflow rejects commits outside `origin/main`, dirty release inputs, mismatched package versions, and missing Apple secrets. These are release-safety checks, not updater failures.

### Notarization fails

Verify that the API key, key ID, and issuer ID belong to the same App Store Connect account and that the Developer ID certificate has not expired or been revoked.

### Private repository

Normal end-user distribution should keep the update repository public. A private GitHub feed requires every installed client to hold a GitHub token, which is inappropriate for builds shared with friends.
